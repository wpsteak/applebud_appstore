# Applebud App Store MVP 實作文件

## 1. 文件目的

本文件用來記錄 MVP 階段的實作方向。

目前已確認：

- MVP 的核心只是便利貼牆
- 後端使用 Google Apps Script
- 資料儲存使用 Google Sheets

本文件重點不是做長期最優架構，而是定義第一版如何用最簡單方式落地。

## 2. 實作原則

MVP 階段的原則如下：

- 先快速做出可用版本
- 先驗證內容分享與瀏覽需求
- 儘量避免把資料模型做成關聯資料庫思維
- 儘量減少工作表數量
- 儘量減少需要即時計算的資料

因此第一版採用以下策略：

- 不存詳細行為 log
- 資料表盡量打扁
- 工作表數量控制在三張內
- 排行所需資訊直接放在貼文主表

## 3. 技術選擇

### 3.1 後端

第一版後端：

- `Google Apps Script`

用途：

- 提供前端呼叫的 API
- 讀寫 Google Sheets
- 驗證登入者身份
- 處理貼文與首頁資料

### 3.2 資料儲存

第一版資料儲存：

- `Google Sheets`

用途：

- 作為 MVP 階段的簡易資料來源

採用原因：

- 開發快
- 維護成本低
- 與 Google 帳號環境相容
- 足以支撐便利貼牆的第一版需求

## 4. 資料設計原則

第一版資料設計不追求正規化，優先考慮：

- 讀取簡單
- 寫入簡單
- API 邏輯簡單

所以不採以下做法：

- 不拆太多表
- 不做複雜關聯
- 不存詳細點閱 log
- 不做需要大量掃描與統計的資料模型

## 5. Google Sheets 結構

第一版預計只使用三張工作表。

### 5.1 `posts`

用途：

- 儲存便利貼牆的主要貼文資料

建議欄位：

- `id`
- `title`
- `summary`
- `content`
- `category_key`
- `author_email`
- `author_name`
- `status`
- `is_featured`
- `view_count`
- `created_at`
- `updated_at`
- `published_at`

說明：

- 這張表是核心主表
- 首頁大部分內容都直接從這張表產生
- `view_count` 為彙總欄位，不是事件 log

### 5.2 `categories`

用途：

- 儲存可選分類

建議欄位：

- `key`
- `name`
- `sort_order`
- `is_active`

說明：

- 分類維持少量即可
- 第一版由管理者維護

### 5.3 `config`

用途：

- 儲存系統級設定

可放內容：

- 管理者 email 清單
- 公司網域設定
- 首頁顯示數量
- 其他少量設定值

說明：

- 這張表的目的，是避免再多拆一張 `admins`
- 如果管理者資訊很簡單，直接放在 `config` 即可

## 6. 關於 `view_count` 的設計

第一版不存詳細點閱 log，改採用彙總欄位：

- `view_count`

這個欄位的定位是：

- 作為排序訊號
- 作為首頁排行依據
- 不作為精準流量分析依據

### 6.1 這樣做的好處

- 不需要另外維護大量 `views` 資料
- 首頁讀取更簡單
- 排行計算更直接
- 可避免工作表快速膨脹

### 6.2 這樣做的限制

- 無法回溯每一次點閱
- 無法精準重算週排行與月排行
- 不適合做深入分析

### 6.3 目前結論

MVP 階段可以接受這個限制。

`view_count` 只要能大致反映內容被看的程度，就足夠支撐首頁排序。

## 7. 首頁區塊實作影響

因為目前不存詳細 log，所以首頁區塊需要跟資料模型一致。

### 7.1 穩定可支撐的區塊

- 精選
- 最新張貼
- 總排行

這三個區塊都可以直接從 `posts` 表取得。

### 7.2 相對不適合第一版硬做的區塊

- 月排行
- 週排行

原因：

- 沒有原始點閱紀錄時，這兩種排行難以精準計算
- 若硬做，反而需要額外欄位與維護邏輯，會破壞 MVP 的簡單性

### 7.3 建議

若要維持第一版簡潔，首頁建議先保留：

- 精選
- 最新張貼
- 總排行

等未來需要更細緻排行時，再重新設計資料來源。

## 8. API 範圍

第一版 Apps Script API 只處理便利貼牆核心需求。

### 8.1 使用者資訊

- 取得目前登入者資訊

用途：

- 判斷是否為公司帳號
- 判斷是否為管理者

### 8.2 貼文

- 取得首頁資料
- 取得貼文列表
- 取得單篇貼文
- 建立貼文
- 編輯貼文
- 刪除貼文

### 8.3 分類

- 取得分類列表

### 8.4 管理功能

- 設定精選
- 取消精選
- 下架貼文

### 8.5 點閱

- 單篇貼文被打開時，更新 `view_count`

## 9. 權限邏輯

第一版權限維持最簡：

- 只有公司 Google 帳號可使用
- 使用者只能編輯自己的貼文
- 使用者只能刪除自己的貼文
- 管理者可設定精選
- 管理者可下架任何貼文

## 10. 效能判斷

以目前這個設計來看，效能風險主要比原先低很多。

原因：

- 沒有大量 log 表
- 首頁主要只讀一張 `posts`
- 排行可直接用 `view_count`
- 幾乎不需要做複雜 join 或統計

### 10.1 `view_count` 會不會成為瓶頸

目前判斷：

- 作為讀取欄位，不太會是瓶頸
- 真正可能出現壓力的是大量併發更新同一欄位

但以內部便利貼牆 MVP 而言，這通常是可接受的。

第一版設計原則：

- `view_count` 只當排序訊號
- 不追求百分之百精準
- 不把它當正式分析資料

## 11. 已知限制

採用目前這種簡化模型，會有以下限制：

- 無法知道誰看過哪篇文章
- 無法做精準週排行與月排行
- 無法做細緻互動分析
- 不適合支撐複雜推薦或報表

但這些都不影響 MVP 要驗證的核心問題。

## 12. 後續升級方向

若未來內容量提升，或需要更細緻分析，可再分階段升級：

1. 導入第三方行為追蹤服務
2. 將排行與分析從 Sheets 拆開
3. 補更細的統計資料來源
4. 再決定是否升級正式資料庫

第一版不先處理這些。

## 13. 目前結論

目前 MVP 的實作結論如下：

- 核心產品是便利貼牆
- 後端使用 Google Apps Script
- 資料儲存使用 Google Sheets
- 工作表先控制在三張內
- 不存詳細行為 log
- 以 `view_count` 作為簡化排序訊號
- 首頁先以精選、最新、總排行為主

這樣的設計足夠簡單，也更符合 MVP 階段的目標。
